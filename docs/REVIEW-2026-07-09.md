# Full project review — pdf-to-md

**Date:** 2026-07-09  
**Scope:** Entire `Desktop/pdf-to-md` tree (library, API, UI, Docker, docs)  
**Verdict:** Solid v0.1 foundation for a local/personal webapp. Architecture is right. Not production-safe without hardening. OCR on bare Windows is incomplete until Tesseract/Poppler (or Docker) are set up.

---

## Summary

The project cleanly separates concerns: `pdf_to_md` (importable converter), `app` (FastAPI + jobs), `static` (UI), and Docker (OCR stack). Hybrid digital-text-then-OCR, async jobs with progress, and health probes match the stated goal of something you can build a webapp on.

Main risks: resource exhaustion (no page/memory limits), open network exposure without auth, Poppler discovery not wired into `pdf2image`, no automated tests, and image extraction quality. Product direction (stay webapp, prefer Docker for OCR) is correct.

## Issues

### Issue 1 — Severity: bug
- File: `pdf_to_md/deps.py` (~108–118) + `pdf_to_md/ocr.py` (`convert_from_path`)
- Description: Poppler may be detected under common Windows paths (`C:\poppler\...`) and health can report `poppler_available: true`, but `pdf2image.convert_from_path` is never given `poppler_path=`. Conversion still fails unless `pdftoppm` is on PATH.
- Suggestion: When probing finds a non-PATH binary, return its directory and pass `poppler_path=` into every `convert_from_path` call (via options or module-level configure).
- Status: open

### Issue 2 — Severity: bug
- File: `app/main.py:38-44`
- Description: `allow_origins=["*"]` with `allow_credentials=True` is invalid per CORS and misleading if you later call the API cross-origin with credentials.
- Suggestion: For local same-origin UI, use `allow_credentials=False` or explicit origins (`http://127.0.0.1:8000`). Never ship `*` + credentials.
- Status: open

### Issue 3 — Severity: bug (ops / security)
- File: `app/jobs.py`, `app/routes.py`
- Description: No auth. Job IDs are unguessable UUIDs, but binding `0.0.0.0` (Docker) exposes upload + result markdown to the LAN. Anyone who obtains a job_id can read the document.
- Suggestion: Document “localhost only” clearly; default Docker publish to `127.0.0.1:8000:8000`; add optional API key later if exposed beyond loopback.
- Status: open

### Issue 4 — Severity: bug (resource / DoS)
- File: `app/routes.py:55-71`, `app/jobs.py:75-88`, `pdf_to_md/ocr.py`
- Description: Entire PDF is read into memory; no max page count; OCR at 300–600 DPI page-by-page can exhaust CPU/RAM; job queue only caps finished job *records* (200), not concurrent upload size × workers. A large scanned PDF is an easy local DoS.
- Suggestion: Cap pages (e.g. 100), cap DPI, optionally stream/spool uploads to temp disk with size limit enforced while writing; reject oversized page counts before OCR.
- Status: open

### Issue 5 — Severity: suggestion
- File: `pdf_to_md/images.py:51-53`
- Description: Embedded streams are always saved as `.png` regardless of actual codec (often JPEG/DCT). Broken image files and wrong markdown links.
- Suggestion: Sniff magic bytes (`%PDF`, `JFIF`, `PNG`, etc.) or use filter names from the PDF; only emit supported formats or convert via Pillow.
- Status: open

### Issue 6 — Severity: suggestion
- File: project-wide
- Description: No automated tests (unit for hybrid thresholds, API for `/jobs` lifecycle, deps probe mocks). Regression risk is high as features grow.
- Suggestion: Add `tests/` with pytest + a tiny digital PDF fixture; mock OCR; TestClient needs `httpx` (current Starlette message may say `httpx2` depending on version).
- Status: open

### Issue 7 — Severity: suggestion
- File: `pdf_to_md/converter.py:118-121`
- Description: On OCR path, `ocr_ready()` and `ocr_not_ready_message()` each call `probe_dependencies()` — double subprocess/which work per conversion.
- Suggestion: Call `probe_dependencies` once; branch on `report.ocr_ready`.
- Status: open

### Issue 8 — Severity: suggestion
- File: `pdf_to_md/markdown.py`, converter output
- Description: Output is page-sliced plain text, not structure-aware Markdown (headings, lists, tables). Fine for v1; weak for study notes / web publishing.
- Suggestion: Document limitation; later optional pymupdf layout or ML tools stay out of core.
- Status: open

### Issue 9 — Severity: suggestion
- File: repo root
- Description: No `.gitignore` (`.venv`, `__pycache__`, temp PDFs). Not a git repo yet either.
- Suggestion: `git init` + `.gitignore` with `.venv/`, `__pycache__/`, `*.pyc`, `.env`, `_test*`.
- Status: open

### Issue 10 — Severity: suggestion
- File: `Dockerfile` + `docker-compose.yml`
- Description: Container runs as root; compose publishes all interfaces. Acceptable for personal use; weak for shared hosts.
- Suggestion: Non-root USER; `ports: ["127.0.0.1:8000:8000"]`.
- Status: open

### Issue 11 — Severity: nit
- File: `pdf_to_md/deps.py:83`
- Description: Helper named `Path_exists` breaks PEP 8 (looks like a class).
- Suggestion: Rename to `path_exists`.
- Status: open

### Issue 12 — Severity: nit
- File: `app/main.py` version vs `pdf_to_md.__version__` vs `pyproject.toml`
- Description: Version strings can drift (FastAPI app hardcodes `"0.1.0"` while health uses package version).
- Suggestion: Single source: `from pdf_to_md import __version__` in `create_app`.
- Status: open

### Issue 13 — Severity: nit
- File: `static/app.js` `setBusy(false)`
- Description: Hides progress and clears poll timer; success path re-shows bar — slightly awkward but works. Cancel-in-flight not supported.
- Suggestion: Optional AbortController / cancel job endpoint later.
- Status: open

## Security / deployment checklist

| Item | Status |
|------|--------|
| Upload size limit | Yes (`PDF_TO_MD_MAX_UPLOAD_MB`) |
| File type check | Extension-only `.pdf` (not magic-byte) |
| Auth | None |
| Rate limit | None |
| Path traversal on filename | Mitigated via `Path(...).name` |
| XSS in UI | Warnings escaped with `escapeHtml` |
| Secrets in repo | None observed |
| Multi-worker job store | Unsafe (documented; Docker uses 1 worker) |
